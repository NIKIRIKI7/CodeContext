# Architecture Decision Records (ADR)

## ADR-001: Реестр редьюсеров вместо if-elif цепочки

**Статус:** Принято  
**Контекст:** `Store._reducer` содержал 60+ строк `if-elif`. Добавление нового action type требовало редактирования этого метода.  
**Решение:** Словарь `{action_type: handler_function}`. Каждый обработчик — отдельный метод.  
**Последствия:** Добавление нового action — только регистрация нового метода в `_register_handlers`. Существующие обработчики не меняются (OCP).

## ADR-002: Use Cases как отдельный слой

**Статус:** Принято  
**Контекст:** Логика `scan`, `process`, `github` была размазана по `MainController`, делая его «Богом-объектом».  
**Решение:** Каждый бизнес-сценарий — отдельный Use Case класс. Контроллер только координирует.  
**Последствия:** Логика переиспользуется между GUI и CLI. Контроллеры стали тонкими (< 100 строк).

## ADR-003: DIContainer как единственная точка сборки

**Статус:** Принято  
**Контекст:** Сервисы создавались в трёх местах: `MainController.__init__`, `CliController.__init__`, `DIContainer` (неиспользуемый). Дублирование кода инициализации.  
**Решение:** `DIContainer` — единственный создатель объектов. `main.py` создаёт контейнер и передаёт объекты через аргументы.  
**Последствия:** Легко тестировать (подменить репозиторий на мок в DIContainer). Нет дублирования.

## ADR-004: `IntegrationService` делегирует стратегии

**Статус:** Принято  
**Контекст:** `integration_strategies.py` реализовывал Windows/Linux/macOS стратегии, но `IntegrationService` дублировал всю Windows-логику инлайн.  
**Решение:** `IntegrationService` принимает `ContextMenuStrategy` через конструктор. `DIContainer` выбирает нужную реализацию по `platform.system()`.  
**Последствия:** Поддержка новой ОС — только новый класс стратегии. `IntegrationService` стал 10-строчным фасадом.

## ADR-005: `Store.state` — только глубокая копия

**Статус:** Принято  
**Контекст:** Если `Store.state` возвращает прямую ссылку, любой код может случайно мутировать состояние в обход reducer.  
**Решение:** `Store.state` всегда возвращает `copy.deepcopy`. Мутации возможны только через `dispatch`.  
**Последствия:** Небольшой overhead на копирование, но гарантия иммутабельности снаружи.

## ADR-006: `winreg` — только в стратегии, не на уровне модуля

**Статус:** Принято  
**Контекст:** `integration_service.py` импортировал `import winreg` на уровне модуля → `ImportError` на Linux/macOS.  
**Решение:** `winreg` импортируется только внутри `WindowsContextMenuStrategy`, защищённый `if platform.system() == "Windows"`.  
**Последствия:** Приложение запускается на всех платформах. Стратегия изолирует платформо-зависимый код.

## ADR-007: Все action types определены в одном файле

**Статус:** Принято  
**Контекст:** `ProcessWorkspaceUseCase` ссылался на `WORKFLOW_STARTED`, `LOG_MESSAGE_ADDED` и другие константы, которые **не существовали** в `action_types.py`. Runtime `NameError`.  
**Решение:** Все action type константы — только в `src/actions/action_types.py`. Запрещено создавать строки-литералы action types вне этого файла.  
**Последствия:** `NameError` выявляются статическими анализаторами, а не в runtime.

## ADR-008: `CliController` и `MainController` используют общие Use Cases

**Статус:** Принято  
**Контекст:** Оба контроллера содержали дублирующуюся логику pipeline: читать → чистить → форматировать → экспортировать.  
**Решение:** Общий `ProcessWorkspaceUseCase` и `ScanWorkspaceUseCase`. Оба контроллера получают их через DI.  
**Последствия:** Логика pipeline в одном месте. Изменение алгоритма — в одном файле.
